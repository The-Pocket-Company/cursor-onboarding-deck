# Karpathy 行為準則（繁中）

降低 LLM 寫程式常見錯誤的行為準則，靈感來自 Andrej Karpathy 對 LLM 寫程式常見陷阱的觀察。

**取捨：** 這些準則偏向「謹慎優先於速度」。瑣碎任務請自行斟酌。

## 1. 動手前先想清楚

**別假設、別藏住疑惑、把取捨講出來。**

實作前：
- 明確說出你的假設；不確定就問。
- 有多種解讀時，全部列出來，別自己默默選一個。
- 有更簡單的做法就直說；該反對時就反對。
- 有不清楚的地方就停下來，指出哪裡不清楚，然後發問。

## 2. 簡單優先

**用最少的程式碼解決問題，不要臆測性的東西。**

- 不做沒被要求的功能。
- 不為一次性的程式碼做抽象。
- 不加沒人要求的「彈性」或「可設定性」。
- 不為不可能發生的情境寫錯誤處理。
- 如果你寫了 200 行、但其實 50 行就夠，請重寫。

問自己：「資深工程師會不會覺得這太複雜？」會的話，就簡化。

## 3. 最小幅度修改

**只動你非動不可的地方；只收拾你自己造成的東西。**

修改既有程式碼時：
- 別「順手改善」旁邊的程式碼、註解或排版。
- 別重構沒壞的東西。
- 配合既有風格，就算你會用不同寫法。
- 看到無關的死碼，提出來就好，別刪。

當你的改動產生了沒用到的東西（孤兒）：
- 移除「因為你的改動」才沒用到的 import／變數／函式。
- 沒被要求，就別刪原本就存在的死碼。

檢驗標準：每一行改動，都要能直接對應到使用者的需求。

## 4. 目標導向執行

**先定義成功標準，再迭代到驗證通過為止。**

把任務轉成可驗證的目標：
- 「加驗證」→「先為無效輸入寫測試，再讓它通過」
- 「修這個 bug」→「先寫一個能重現它的測試，再讓它通過」
- 「重構 X」→「確保重構前後測試都通過」

多步驟任務，先給一份簡短計畫：
```
1. [步驟] → 驗證：[檢查]
2. [步驟] → 驗證：[檢查]
3. [步驟] → 驗證：[檢查]
```

夠強的成功標準能讓你獨立迭代；薄弱的標準（「能動就好」）會逼你不斷來回確認。

---

**這些準則奏效的徵兆：** diff 裡多餘的改動變少、因過度複雜而重寫的次數變少，而且「釐清的提問」出現在犯錯之前，而不是之後。
